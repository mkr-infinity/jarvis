3. API KEY HANDLING - SECURITY AUDIT
================================================================================

STORAGE FLOW:
  Frontend Input → WebSocket JSON → SQLite settings table
  
CURRENT IMPLEMENTATION:
  Frontend: /renderer/scripts/app.js lines 57-60, 351-362
  Backend: /backend/app/services/database.py lines 24-27, 76-79
  Usage: /backend/app/services/ai_engine.py lines 64-149

KEYS MANAGED:
  ✓ OpenAI (uses Bearer token in header)
  ✓ Anthropic (uses x-api-key header)
  ⚠️ Gemini (CRITICAL: puts key in URL - highest risk!)
  ✓ Groq (OpenAI-compatible, header-based)

SECURITY VULNERABILITIES IDENTIFIED:

HIGH SEVERITY:
  🔴 API Keys stored in PLAIN TEXT in SQLite
     → No encryption at rest
     → Database readable by any process
     → Location: ~/.config/J.A.R.V.I.S/data/jarvis.db
  
  🔴 Gemini API key in URL (line 149)
     → Logged in browser history, server logs, crash dumps
     → Highest exposure risk
  
  🔴 Keys transmitted in WebSocket JSON (localhost, but unencrypted)
     → Could be intercepted with network snooping tools

MEDIUM SEVERITY:
  🟠 No key validation before storage
     → User can save invalid keys
     → No test-before-save mechanism
  
  🟠 Settings export includes all keys (database.py line 200-203)
     → If bootstrap ever exposed, all credentials leaked
  
  🟠 No key masking in UI
     → Settings display shows full keys in DOM
     → If settings logged, keys exposed

LOW SEVERITY:
  🟡 No key rotation/expiry reminders
  🟡 No rate limiting on key-based requests
  🟡 No audit log for API key access
  🟡 No timeout to clear keys from memory

ERROR HANDLING FOR KEYS:
  Frontend: Only validates if empty (line 426-429)
  Backend: Generic message if key missing (ai_engine.py line 70-72)
  No: Authentication errors, rate limit errors, validation

IMMEDIATE FIXES REQUIRED:
  1. Encrypt keys with SQLcipher or application-level encryption
  2. Move Gemini key from URL to header (use Authorization header)
  3. Add key validation before storage
  4. Mask keys in settings display (show last 4 chars only)
  5. Filter sensitive data from bootstrap export
  6. Add key expiry warnings

MEDIUM-TERM IMPROVEMENTS:
  1. Use OS keychain (Keychain on macOS, Credential Manager on Windows)
  2. Implement key rotation scheduling
  3. Add rate limiting and usage alerts
  4. Create audit log for API key access
  5. Add connectivity test before accepting keys

LONG-TERM ARCHITECTURE:
  1. Consider remote credential management
  2. OAuth2 flows for providers
  3. Encrypted sync to cloud provider
